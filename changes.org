** semantic
- restored --lsp flag (was semgrep-core -lsp in 2020) to connect to
  an LSP server (e.g. ocamllsp)
  and get inferred type information for identifiers during pattern matching,
  enabling typed metavariables like ($X : string) to work without
  type annotations in the source code. A similar feature was
  added in Q3 2020 as a POC (Proof Of Concept).
  The --lsp flag is designed as a generic channel to LSP servers: typing
  is the first use case, but name resolution (fully qualified names,
  alias resolution via textDocument/definition) is a natural next step.
- new --lsp-expr flag (implies --lsp) extends this to arbitrary
  expressions (function calls, match expressions, etc.), not just
  identifiers. This can slow down a lot the semgrep query though.
- --lsp now supports C via clangd (in addition to OCaml via ocamllsp).
  The language server is selected automatically from the --lang flag.
  clangd resolves types across files (via headers), enabling interfile
  typed metavar matching like (int $X) on function call return values.
- --lsp now supports C++ via clangd (same server as C). clangd resolves
  class method return types, template instantiation types, and 'auto'
  deduction across files (7 findings without LSP → 18 with LSP).
- --lsp now supports Go via gopls. gopls resolves interfile types,
  e.g. matching ($X: int) on variables whose type comes from a function
  defined in another file (6 findings without LSP → 10 with LSP).
- --lsp now supports Rust via rust-analyzer. rust-analyzer resolves
  interfile types including trait impls and generic instantiations
  (2 findings without LSP → 8 with LSP).
- --lsp now supports Python via ty (Astral's fast Rust type checker)
  or pyright (Microsoft). Both servers resolve interfile types via
  type inference.
- --lsp now supports TypeScript and JavaScript via typescript-language-server.
  Both languages use the same server (like C/C++ share clangd). The server
  resolves interfile types including cross-module imports
  (5 findings without LSP → 9 with LSP).
  Note: typed metavar matching requires --lang ts (Lang.Js has no typed
  metavar support in Parse_metavariable_type.ml).
- --lsp now supports Java via jdtls (Eclipse JDT Language Server).
  jdtls resolves variable types across files via classpath/Maven project
  configuration. Note: jdtls hover only returns types for identifiers,
  not for method call expressions.
- --lsp now supports C# via OmniSharp (Roslyn-based language server).
  OmniSharp resolves interfile types via Roslyn, enabling typed metavar
  matching like (int $X) on variables and expressions whose types
  are defined in other files.
- --lsp now supports Scala via Metals. Metals resolves interfile types
  via the Scala presentation compiler, enabling typed metavar matching
  like ($X: Int) on variables and expressions whose types are defined
  in other files.
- --lsp now supports PHP via Intelephense. Intelephense resolves interfile
  types including class method return types and variable type inference.
  PHP typed metavars use ($X : int) syntax (colon-separated, like TypeScript)
  to avoid ambiguity with PHP cast expressions like (int) $x.
- --lsp now supports Haskell via HLS (Haskell Language Server). HLS resolves
  interfile types including type class instances and type inference.
  Haskell typed metavars use ($X :: Int) syntax (double-colon, matching
  Haskell's native type annotation operator).
  Limitation: the tree-sitter grammar cannot parse ($X :: Int) patterns
  because :: is ambiguous with the list cons operator when $X is a metavar.
  The LSP hover parsing infrastructure is ready but TypedMetavar patterns
  require a grammar fix. The metavariable-type: rule key works for static
  type matching.
- added test data in tests/lsp/{ocaml,c,cpp,go,rust,python,typescript,java,csharp,scala,php,haskell}/
  demonstrating interfile type resolution with --lsp for each language.

** LSP infrastructure
- auto-detection of the project root for LSP servers: find_project_root
  walks upward from the scanning root looking for a marker file
  (go.mod for Go, dune-project for OCaml, compile_commands.json for C,
  Cargo.toml for Rust, pyproject.toml for Python, tsconfig.json for TS,
  pom.xml for Java, *.csproj for C#, build.sbt for Scala, composer.json for PHP). Markers can now use glob patterns
  (e.g. "*.csproj") for languages with variable project file names.
  This also fixes workspace URI handling for gopls
  which requires absolute file:// URIs matching the workspace root.
- LSP_client now responds to server-initiated requests (e.g.
  window/workDoneProgress/create) instead of silently ignoring them.
- warmup support for servers that need time to index after initialization:
  drain $/progress tokens (rust-analyzer), language/status ServiceReady
  (jdtls), and idle-timeout after didOpen (typescript-language-server).
- hover response handling for all three LSP formats: MarkupContent
  (most servers), MarkedString (single), and MarkedString list (jdtls).
- per-language hover parsing: each LSP_<lang>.ml module extracts the type
  string from the server's hover format (code fences, prefixes, signatures)
  and parses it into AST_generic.type_ via the language's menhir parser.
- moved LSP_lang.t (shared type) to libs/lib_languages/ and each
  LSP_<lang>.ml to languages/<lang>/lsp/ (as parser_<lang>.lsp sub-libraries),
  so per-language LSP helpers are reusable by other projects (efuns, codemap)
  without pulling in osemgrep-specific code.

** languages
- add Haskell language support (alpha): tree-sitter parser via semgrep-haskell
  submodule, intermediate AST_haskell.ml, Haskell_to_generic.ml converter,
  and full CST-to-AST mapping (~1700 lines). Working patterns include type
  signatures ($X :: $Y), data declarations, type aliases, and imports.
- add Zig language support (alpha): menhir parser (ocamllex + menhir),
  AST_zig.ml, Zig_to_generic.ml. Parses basic declarations (const/var),
  functions, field access chains, builtins (@import), string literals,
  anonymous struct/array init (.{...}). More features to be added iteratively.
